# A2팀 사용성 테스트 시나리오
> 작성일: 2026-06-25 | 작성자: 이슬기 (PM)
> 진행 시점: 2주차 3일차 (이미지 생성 + 라이브러리 저장 완성 후)

---

## 1. 테스트 목적

**핵심 질문:** 방송 제작 배경이 있는 비개발자가 5분 안에 KX GenStudio로 이미지 1장을 생성하고 저장할 수 있는가?

**확인할 것:**
- 프롬프트 입력창을 처음 보고 무엇을 해야 할지 알았는가
- 막힌 지점이 어디였는가
- 생성 결과물에 대한 첫 반응
- 라이브러리 저장이 직관적인가

---

## 2. 참여자 기준

| 항목 | 기준 |
|------|------|
| 인원 | 1~2명 |
| AI 이미지 생성 경험 | 없거나 초보 (Midjourney·ChatGPT 이미지 사용 1~2회 이하) |
| 방송·영상 배경 | 있으면 우대 (없어도 무방) |
| 비개발자 | 필수 — 개발자 시각으로 사용성 왜곡되지 않도록 |

**후보 탐색:** 팀원 지인, SNS, 방송영상 전공 동기 등

---

## 3. 테스트 전 준비

### PM이 준비할 것

- [ ] 테스트할 URL 확인 (로컬 or 스테이징 배포)
- [ ] 테스트용 API 키 잔액 확인 (최소 이미지 5~10장 생성 가능한지)
- [ ] 관찰 기록 시트 출력 or 노션 준비
- [ ] 화면 녹화 툴 준비 (Loom, QuickTime 등)
- [ ] 참여자에게 사전 안내 문자 발송

### 참여자에게 사전 안내 (테스트 전날)

```
안녕하세요! 인턴십 프로젝트 사용성 테스트에 참여해 주셔서 감사해요.

내일 테스트는 약 20~30분 소요되고, 방법은 간단해요:
- 제가 만든 서비스에서 이미지를 직접 만들어 보시면 됩니다
- 막히는 곳이 생기면 말씀해 주세요 (해결해 드리는 게 아니라 관찰하는 거예요)
- 틀린 게 없어요. 어색하거나 불편한 게 있으면 솔직하게 말씀해 주세요.
- 서비스를 평가하는 테스트이지, 참여자분을 평가하는 게 아닙니다.

[링크] ← 테스트 당일 공유 예정
```

## 4. 테스트 태스크

> **진행 방식:** 태스크를 구두로 말해주고 참여자가 직접 진행. PM은 옆에서 조용히 관찰·기록. 막혀도 힌트 주지 않음 (단, 5분 이상 완전히 막히면 다음 태스크로 이동).

---

### 태스크 1 — 이미지 생성 (메인)

**소요 목표: 5분 이내**

말해줄 내용:
> "방송 드라마 촬영 전에 장면을 미리 시각화하고 싶어요. 빗속 골목에서 우산을 들고 혼자 걷고 있는 주인공의 이미지를 만들어 보세요. 어떤 느낌이어야 하는지는 자유입니다."

**관찰 포인트:**

| 관찰 항목 | 기록 내용 |
|---------|---------|
| 프롬프트 입력 언어 | 한국어로 입력했는가, 영어로 입력했는가 |
| 모델 선택 UI 인지 | 모델 선택 드롭다운을 발견했는가 (발견 시간: ___초) |
| 비율 옵션 인지 | 비율 옵션을 인지했는가 |
| 생성 버튼 위치 | 버튼을 바로 찾았는가, 헤맸는가 |
| 로딩 중 반응 | 기다리는 동안 무슨 말/행동을 했는가 |
| 결과 첫 반응 | 생성된 이미지를 보고 처음 한 말 |
| 재생성 여부 | 결과가 마음에 안 들어서 다시 했는가 |
| 소요 시간 | ___분 ___초 |
| 성공 여부 | 성공 / 부분 성공 / 실패 |

---

### 태스크 2 — 라이브러리 저장

**소요 목표: 2분 이내**

말해줄 내용:
> "마음에 드는 이미지를 보관함(라이브러리)에 저장해 보세요."

**관찰 포인트:**

| 관찰 항목 | 기록 내용 |
|---------|---------|
| 저장 버튼 발견 시간 | ___초 |
| 저장 방법 인지 | 클릭 한 번에 저장했는가, 헤맸는가 |
| 저장 완료 피드백 | 저장됐다는 것을 어떻게 확인했는가 |
| 라이브러리 탭 이동 | 스스로 라이브러리로 이동했는가 |
| 소요 시간 | ___분 ___초 |

---

### 태스크 3 — 다운로드 (선택)

말해줄 내용:
> "저장된 이미지를 파일로 받아볼 수 있는지 확인해 보세요."

**관찰 포인트:**

| 관찰 항목 | 기록 내용 |
|---------|---------|
| 다운로드 버튼 발견 | 발견 시간 ___초 |
| 파일 저장 성공 | 성공 / 실패 |

## 5. 테스트 후 인터뷰 (5분)

태스크 완료 후 아래 질문을 순서대로 물어보세요.

| # | 질문 | 목적 |
|---|------|------|
| 1 | "처음 화면을 봤을 때 이게 뭐하는 서비스인지 바로 알았나요?" | 첫인상·서비스 이해도 |
| 2 | "프롬프트 입력창에 뭘 써야 할지 바로 알았나요? 어떻게 생각하고 입력했나요?" | 프롬프트 진입장벽 |
| 3 | "이미지 생성되는 걸 기다리는 동안 어땠나요?" | 로딩 UX |
| 4 | "결과 이미지가 원하던 것과 얼마나 비슷했나요? (1~5점)" | 결과 만족도 |
| 5 | "가장 불편하거나 헷갈렸던 부분은 어디였나요?" | 주요 통증 지점 |
| 6 | "방송 현장에서 실제로 쓴다면 어떤 기능이 더 있었으면 좋겠나요?" | 미충족 니즈 발굴 |

## 7. 피드백 반영 기준 (2주차 4일차)

사용성 테스트 결과를 아래 기준으로 처리합니다.

| 유형 | 기준 | 처리 |
|------|------|------|
| 즉시 수정 | 2명 중 2명이 막힌 지점 | 당일 FE에게 전달, 이번 주 안에 반영 |
| 다음 주 반영 | 1명이 막혔거나 사소한 표현 문제 | 백로그 등록, 3주차에 반영 |
| 백로그 보류 | 좋은 아이디어지만 MVP 범위 외 | 백로그 등록, 추후 고도화 검토 |
| 기각 | 소수 의견이고 현재 설계에 이유 있음 | 기각 사유 기록 |
