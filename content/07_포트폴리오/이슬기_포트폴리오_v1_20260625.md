# [슬라이드 05] 핵심 판단 ① — 기능 범위 경계를 어떻게 그을 것인가

## 상황

킥오프 회의록에 "A-B팀 경계에 혼선이 있었다"고 명시되어 있었다.
명세서 상 기능 목록은 있지만, 어느 팀이 무엇을 만드는지 합의된 기준이 없었다.

## 고민

```
스토리보드 그리드 기능이 어디에 속하는가?
  - A팀이 이미지를 생성하므로 → A팀이 9컷 그리드 이미지도 만들어야 하는가?
  - B팀이 이미지 분석(→ 텍스트)을 하므로 → B팀이 그리드도 포함인가?
```

초기에는 PD의 발언("9컷 아웃컷 이미지 + 샷별 프롬프트")을 근거로 A2팀 범위로 추가 정의했다.
이후 B팀 Q&A와 역할 재확인을 통해 스토리보드 그리드는 B팀의 핵심 기능임을 확정.

## 판단

**F002 스토리보드 그리드 → B팀 범위. A2팀 문서 전량에서 제거.**

- 7개 문서 일괄 수정 (기획서·PRD·마일스톤·BE기술기획·QA체크리스트·아젠다·포트폴리오)
- A-B 연동 방식도 확정: 자동 API 연결 없음, 사용자가 로컬 저장 후 B팀 서비스에 직접 업로드

## 왜 이 결정이 중요했는가

> 잘못된 범위로 코딩을 시작하면 → 3주차에 버릴 코드가 생긴다
> PM이 킥오프 전에 경계를 명확히 잡아놓으면 → 팀은 첫 날부터 맞는 방향으로 일한다

# [슬라이드 06] 핵심 판단 ② — 이미지 모델을 어떻게 쓸 것인가

## 상황

명세서에는 이미지 생성 API로 Gemini 단일 사용이 명시되어 있었다.
그런데 GPT Image 2가 2025년 말 출시 후 품질 측면에서 비교 우위가 명확했다.

## 고민

```
GPT Image 2를 쓰고 싶다. 그런데:
- 가격: GPT Image 2 = $0.07~0.167/장 vs Gemini = $0.039/장
- 속도: GPT Image 2 = 1분 내외 vs Gemini = 12초
- 팀원 60시간 테스트 기간 동안 GPT Image 2만 쓰면 예산이 부족해질 수 있다
```

## 판단

**개발·테스트 = Gemini 2.5 Flash / 데모·최종 발표 = GPT Image 2 이원화 전략**

| 상황 | 사용 모델 | 이유 |
|------|---------|------|
| 개발 중 반복 테스트 | Gemini 2.5 Flash | 속도 빠름, 비용 55% 절감 |
| 클라이언트 데모, 최종 발표 | GPT Image 2 | 품질 최우선, PD 설득력 |

## 결과

이 전략으로 이미지 생성 API 비용을 $39.82로 통제 가능.
FE에서는 모델 선택 드롭다운 하나로 두 모델 전환이 가능하도록 요청해 구현 복잡도 최소화.

# [슬라이드 07] 핵심 판단 ③ — 비용 리스크를 어떻게 막을 것인가

## 상황

영상 생성 API(fal.ai)는 클립 1개당 $0.55(Kling 5초) ~ $3.00(Seedance 10초).
팀원 6명이 1개월간 기능 테스트를 하면 예산이 가장 먼저 소진되는 영역.

## 고민

```
API 예산 $230 중 영상이 약 65%($150)를 차지하는 구조
  - 팀원이 영상 테스트를 무제한으로 하면 → 2주차에 예산 소진
  - 그렇다고 테스트를 제한하면 → 영상 품질 검증을 못함
```

## 판단

두 가지 통제 장치를 동시에 설계

**① 모델·길이 분리 정책**
```
개발·테스트: Kling 3.0, 5초 → $0.55/클립 (Seedance의 약 1/5)
데모·최종:  Seedance 2.0, 10초 → $3.00/클립
```

**② fal.ai Hard Limit $120 설정 → 킥오프(6/30) Day 1 의무 액션으로 지정**

> 단순 "규칙"이 아니라, BE가 첫날 완료해야 하는 체크리스트 항목으로 마일스톤에 내장

## 왜 Day 1인가

예산 초과는 발생 후 인지하면 이미 늦다. 한도 설정은 코딩 시작 전에 해야 의미 있다.

# [슬라이드 08] 핵심 판단 ④ — 역프롬프트 API를 어떻게 설계할 것인가

## 상황

역프롬프트(F004)는 레퍼런스 이미지를 분석해 재현 가능한 영문 프롬프트를 추출하는 기능.
초기 설계에서는 Gemini 2.5 Flash의 멀티모달 기능을 사용하려 했다.

## 고민

```
이미 Gemini를 이미지 생성 테스트에 쓰고 있다.
그런데 역프롬프트는 "영문 프롬프트 생성"이 핵심 — 언어 생성 품질이 중요하다.
Claude Sonnet은 프롬프트 생성 정확도가 높고, 이미 Anthropic 키를 쓰기로 했다.

→ 같은 역할(언어 생성)을 두 모델로 나누면 일관성이 떨어지고 관리 포인트가 늘어난다
```

## 판단

**역프롬프트 → Claude Sonnet으로 단일화**

출력 포맷을 JSON으로 강제: `analysis.{style, lighting, composition, lens, color_palette, mood}` + `prompt_en`
- 분석 항목을 태그 형태로 시각화 → 사용자가 "어떤 요소가 추출됐는지" 즉시 파악
- `prompt_en`은 복사 버튼 + [이미지 생성으로 이어서] 버튼과 함께 표시 → F001과 자연스럽게 연결

## 결과로 얻은 것

프롬프트 생성 계열 작업(역프롬프트)을 Claude 단일 API로 통일해 → 호출 로직 단순화, 출력 품질 일관성 확보

# [슬라이드 09] 핵심 판단 ⑤ — 마일스톤을 어떻게 설계할 것인가

## 상황

명세서에 제공된 마일스톤은 기능 목록 나열 수준.
6명이 4주간 비대면으로 일하려면, 각자 "이번 주 내가 할 것"이 명확하게 보여야 한다.

## 고민

```
일반적인 마일스톤: "2주차 — 이미지 생성 완성"
                   → BE가 읽으면? FE가 읽으면? 각자 해석이 달라진다

팀이 필요한 것: 직무별로 "내가 언제 무엇을 완료해야 하는지"가 보이는 마일스톤
```

## 판단

마일스톤을 **일차별 × 직무별 표**로 설계

```
| 일차 | PM | BE | FE | UX/UI |
|------|----|----|----|----|
| 1일  | 기획서 공유, 킥오프 진행 | API 키 발급, 레포 세팅 | 보일러플레이트 세팅 | 레퍼런스 리서치 |
...
```

추가로 내장한 것:
- **등급 기준 (S/A/B)**: 4주차 종료 시점에 무엇이 완료돼야 어느 등급인지 팀이 공유
- **Day 1 의무 액션**: fal.ai Hard Limit 설정 (비용 통제)
- **3주차 판단 포인트**: F004(역프롬프트) 착수 여부를 PM이 당시 진도 보고 판단

## 설계 원칙

> "마일스톤은 PM이 관리하는 문서가 아니라, 팀원이 매일 아침 펼쳐보는 문서다."

# [슬라이드 10] 기술 이해도 — PM이 이해하고 설계에 반영한 것들

> PM이 직접 코드를 짜지는 않지만, 기술을 이해해야 현실적인 기획이 나온다.

| 개념 | PM이 이해한 수준 | 기획에 반영한 방식 |
|------|--------------|----------------|
| fal.ai 비동기 처리 (queue → polling → result) | 흐름 이해, 문서화 | 영상 생성 UX 상태 정의 (IN_QUEUE·IN_PROGRESS·COMPLETED·ERROR), 가짜 프로그레스 바 요청 |
| Supabase Storage vs DB 역할 분리 | 개념 이해 | 이미지 파일 → Storage / 메타데이터 → DB 구조로 스키마 초안 작성 |
| Claude 멀티모달 API | 입출력 구조 이해 | 역프롬프트 시스템 프롬프트 v1 직접 설계, JSON 출력 포맷 정의 |
| 환경변수 보안 (NEXT_PUBLIC_ 규칙) | 규칙 이해 | `SUPABASE_SERVICE_ROLE_KEY`에 NEXT_PUBLIC_ 접두사 금지 → PRD에 명시 |
| API 비용 구조 (토큰·이미지 장수·영상 초 단위) | 산정 가능 | $230 예산 산출, 모델·길이 분리 정책 설계 |

**킥오프 전에 경계를 잡는 것이 PM의 일이다**
