## 2. 공통 명세 — 전 기능 적용

### 2.1 API 응답 표준

```typescript
// 성공
{ success: true, data: { ... } }

// 실패
{ success: false, error: { code: string, message: string } }
```

- `error.message`는 사용자에게 그대로 노출 가능한 한국어 문장으로 반환
- FE는 `error.code` 기준으로 분기하고, 메시지는 `error.message`를 우선 사용

### 2.2 에러 코드 표 (BE·FE 공통)

| 코드 | HTTP | 원인 | 사용자 표시 메시지 |
|------|------|------|------------------|
| `INVALID_INPUT` | 400 | 빈 프롬프트, 잘못된 파라미터 | "입력값을 확인해 주세요." |
| `AUTH_FAILED` | 401 | 이메일·비밀번호 불일치 | "이메일 또는 비밀번호를 확인해 주세요." |
| `SESSION_EXPIRED` | 401 | 세션·토큰 만료 | "로그인이 만료되었습니다. 다시 로그인해 주세요." |
| `DUPLICATE_EMAIL` | 409 | 회원가입 이메일 중복 | "이미 사용 중인 이메일입니다." |
| `FILE_TOO_LARGE` | 400 | 업로드 파일 10MB 초과 | "10MB 이하 파일만 업로드 가능합니다." |
| `UNSUPPORTED_FORMAT` | 400 | 지원하지 않는 파일 형식 | "jpg, png, webp 파일만 업로드 가능합니다." (확인 필요 — §11-8) |
| `GENERATION_FAILED` | 500 | 이미지 생성 실패 | "이미지 생성에 실패했습니다. 다시 시도해 주세요." |
| `VIDEO_GENERATION_FAILED` | 500 | 영상 생성 실패 | "영상 생성에 실패했습니다. 다시 시도해 주세요." |
| `ANALYSIS_FAILED` | 500 | 역프롬프트 분석 실패 | "이미지 분석에 실패했습니다. 다시 시도해 주세요." |
| `TIMEOUT` | 504 | 생성 시간 초과 | "생성 시간이 초과되었습니다. 다시 시도해 주세요." |
| `API_LIMIT_EXCEEDED` | 429 | API 비용 한도 초과 | "현재 사용량 한도에 도달했습니다. 잠시 후 다시 시도해 주세요." |
| `STORAGE_FAILED` | 500 | 저장소 저장 실패 | "저장 중 오류가 발생했습니다. 다시 시도해 주세요." |
| `UNKNOWN` | 500 | 예상치 못한 에러 | "알 수 없는 오류가 발생했습니다. 다시 시도해 주세요." |

> PRD v1 에러 코드 표를 기준으로 AUTH 계열 3종(`AUTH_FAILED`·`SESSION_EXPIRED`·`DUPLICATE_EMAIL`)과 `ANALYSIS_FAILED` 추가 (제안).

### 2.3 비동기 상태 처리 원칙

모든 비동기 동작(생성·저장·조회·업로드)은 아래 3가지 상태를 반드시 구현한다.

| 상태 | UI 기준 |
|------|--------|
| **Loading** | 스피너 또는 스켈레톤 + 트리거 버튼 비활성화. 2초 이상 걸리는 작업은 진행 문구 병기 |
| **Error** | 한국어 에러 메시지 + [다시 시도] 버튼. 에러 코드는 콘솔 로그로만 기록 |
| **Empty** | 안내 문구 + 해당 기능 시작 CTA 버튼 |

### 2.4 중복 요청 방지 규칙 (제안)

- 생성·저장·삭제 등 쓰기 요청은 **응답 수신 전까지 트리거 버튼 비활성화**
- 이미지·영상 생성은 화면당 **동시 1건**만 허용 — 진행 중 재요청 불가
- 폴링 중 화면 이탈 시 폴링 중단 (라이브러리 저장 여부는 §6.2 참조)

### 2.5 토스트·확인 모달 규칙 (제안)

| 상황 | UI |
|------|----|
| 성공 피드백 (저장·복사·변경 완료) | 토스트 3초 자동 닫힘 |
| 복구 불가 동작 (삭제·탈퇴) | 확인 모달 — [취소]가 기본 포커스, 확인 버튼은 위험색 |
| 실패 피드백 | 해당 영역 인라인 에러 메시지 (토스트 지양) |

### 2.6 세션 만료 공통 처리 (제안)

- 어떤 API든 `SESSION_EXPIRED`(401) 수신 시: 토스트 노출 후 로그인 화면으로 이동
- 로그인 성공 시 홈 화면으로 이동 (만료 직전 화면 복원은 이번 범위 제외)

### 2.7 파일명 규칙 (제안)

| 대상 | 규칙 | 예시 |
|------|------|------|
| 이미지 다운로드 | `image_{YYYYMMDD}_{HHmmss}.png` | `image_20260715_143012.png` |
| 영상 다운로드 | `video_{YYYYMMDD}_{HHmmss}.mp4` | `video_20260716_101530.mp4` |

## 3. 로그인·회원가입 (AUTH-001~002)

> PRD v1에서는 인증이 Out of Scope였으나, 킥오프 이후 IA v0.2·기능명세서 v0.1에서 기본 인증 포함으로 변경됨. PRD 업데이트 필요 (§11-1).
> 인증 저장 방식(세션 쿠키 / JWT)·토큰 만료 기준은 BE 확정 (BE 확정).

### 3-1. 회원가입 (AUTH-001)

| 항목 | 내용 |
|------|------|
| 기능 ID | AUTH-001 |
| 화면 | 회원가입 화면 |
| API | `POST /auth/signup` (BE 확정) |

**UI 구성 요소**

| 요소 | 사양 |
|------|------|
| 이메일 입력 | text 필드, placeholder "이메일을 입력해 주세요" |
| 비밀번호 입력 | password 필드, 표시/숨김 토글 아이콘 (제안) |
| 비밀번호 확인 입력 | password 필드, 표시/숨김 토글 아이콘 (제안) |
| [가입하기] 버튼 | 필수 입력 3개가 모두 채워지기 전까지 비활성화 |
| 로그인 화면 이동 링크 | "이미 계정이 있으신가요? 로그인" |

**입력 필드 제약**

| 필드 | 필수 | 제약 | 검증 시점 |
|------|------|------|---------|
| 이메일 | ✅ | 이메일 형식 (`이름@도메인.tld`), 최대 254자, 앞뒤 공백 자동 제거 | 포커스 아웃 + 제출 시 |
| 비밀번호 | ✅ | 8자 이상, 최대 64자 (제안) | 포커스 아웃 + 제출 시 |
| 비밀번호 확인 | ✅ | 비밀번호와 정확히 일치 | 포커스 아웃 + 제출 시 |

**시스템 동작**

1. 클라이언트 검증 통과 확인 (형식·길이·일치)
2. `POST /auth/signup` 호출 — 요청 중 버튼 비활성화 + 스피너
3. 서버에서 이메일 중복 확인
4. 계정 생성 완료 → "가입이 완료되었습니다" 토스트 → 로그인 화면으로 이동 (이메일 값 유지 채움 (제안))
5. 실패 시 해당 필드 하단 인라인 에러 표시

**에러 처리**

| 케이스 | 코드 | 표시 | UI 동작 |
|--------|------|------|--------|
| 이메일 형식 오류 | `INVALID_INPUT` | "올바른 이메일 형식을 입력해주세요" | 이메일 필드 하단 인라인 |
| 비밀번호 8자 미만 | `INVALID_INPUT` | "비밀번호는 8자 이상이어야 합니다" | 비밀번호 필드 하단 인라인 |
| 비밀번호 불일치 | `INVALID_INPUT` | "비밀번호가 일치하지 않습니다" | 확인 필드 하단 인라인 |
| 중복 이메일 | `DUPLICATE_EMAIL` | "이미 사용 중인 이메일입니다" | 이메일 필드 하단 인라인 + 로그인 링크 강조 (제안) |
| 서버 오류 | `UNKNOWN` | 공통 메시지 | 폼 상단 인라인 + 재시도 가능 |

**인수 조건**
- [ ] 유효한 이메일·비밀번호(8자 이상)·일치하는 확인값 입력 시 계정이 생성된다
- [ ] 가입 완료 후 로그인 화면으로 이동한다
- [ ] 중복 이메일로 가입 시도 시 에러 메시지가 표시되고 계정이 생성되지 않는다
- [ ] 필수값 미입력 상태에서는 [가입하기] 버튼이 비활성화된다
- [ ] 요청 중 [가입하기]를 연타해도 요청이 1회만 발생한다

**엣지 케이스**
- 이미 로그인된 상태에서 회원가입 화면 접근 → 홈 화면으로 리다이렉트 (제안)
- 비밀번호에 공백만 8자 입력 → 앞뒤 공백 제거 없이 그대로 허용할지 여부 (확인 필요 — BE와 합의)

### 3-2. 로그인 (AUTH-002)

| 항목 | 내용 |
|------|------|
| 기능 ID | AUTH-002 |
| 화면 | 로그인 화면 |
| API | `POST /auth/login` (BE 확정) |

**UI 구성 요소**

| 요소 | 사양 |
|------|------|
| 이메일 입력 | text 필드 |
| 비밀번호 입력 | password 필드, 표시/숨김 토글 (제안) |
| [로그인] 버튼 | 두 필드 모두 입력 전 비활성화 |

**시스템 동작**

1. `POST /auth/login` 호출 — 요청 중 버튼 비활성화 + 스피너

**에러 처리**

| 케이스 | 코드 | 표시 |

## 4. 이미지 생성 (M001~M006)

> IA v0.5 기준 이미지 생성은 **메뉴 A(초기 레퍼런스, M001~M006 그대로 적용) → 메뉴 B(키이미지, 이하 동일 ID 재사용)** 2단계로 진행된다. 마일스톤 v3.2는 아직 메뉴 A/B 분리 이전 ID 체계라 M001~M006 자체를 나누지 않고, 각 기능 안에서 메뉴 A·메뉴 B 흐름을 구분해 서술한다 (마일스톤 쪽 반영은 별도 후속).

### 4-1. 이미지 생성 화면·프롬프트 입력 (M001)

| 항목 | 내용 |

| 기능 ID | M001 |
| 화면 | 이미지 생성 화면 — 메뉴 A / 메뉴 B 탭 또는 스텝 전환 |
| 담당 | FE + UX/UI (화면) / 이미지 담당 BE (API) |

**공통 — AI 프롬프트 변환 토글** (메뉴 A·B 공통, IA v0.5 확정)

| 요소 | 사양 |
|------|------|
| AI 프롬프트 변환 토글 | On/Off, 기본값 Off (제안) |

**메뉴 A — UI 구성 요소**

| 요소 | 사양 |
|------|------|
| 프롬프트 입력창 | textarea, placeholder "생성하고 싶은 장면을 입력해 주세요" (제안), 자동 높이 확장, 글자 수 카운터 표시 (제안) |

### 4-2. 이미지 생성 호출 (M002)

| 항목 | 내용 |
|------|------|
| 기능 ID | M002 |
| API | `POST /generate-image` |
| 파라미터 | `prompt`, `model: 'gpt-image-2'`(고정), `menu: 'A'|'B'`, `aspectRatio`(메뉴 A), `quality`(메뉴 A), `attributeSelections`(메뉴 B 사이드바 경로 시), `promptConversion: boolean` |

**시스템 동작**

1. 클라이언트 검증: 빈값·공백만·최대 길이 초과 차단
2. AI 프롬프트 변환 토글 On 시, 서버에서 Claude Sonnet 변환 실행 후 변환된 영어 프롬프트로 GPT Image 2 호출 (FE에 변환 결과 노출 없음)
3. `POST /generate-image` 호출 — 버튼 비활성화 + 결과 영역 Loading (스켈레톤 + "이미지를 생성하고 있습니다…")
4. 클라이언트 타임아웃 60초 — 초과 시 요청 중단 후 에러 처리
5. 성공 → 결과 영역에 이미지 표시 (M003) — 메뉴 A는 배경 1장+캐릭터 다각도, 메뉴 B는 캐릭터 키이미지 1장
6. 생성 중 화면 이탈 시 요청 결과는 폐기 (재진입 시 복원 없음) (제안)

**에러 처리**

| 케이스 | 코드 | UI 동작 |
|--------|------|--------|
| 프롬프트 빈값 | — (클라이언트 차단) | 버튼 비활성 + 입력창 하단 "프롬프트를 입력해주세요" |
| 생성 실패 | `GENERATION_FAILED` | 결과 영역 에러 상태 + [다시 시도] — 클릭 시 동일 입력값으로 재호출 |
| 60초 타임아웃 | `TIMEOUT` | 동일 |
| 비용 한도 초과 | `API_LIMIT_EXCEEDED` | 에러 메시지 노출, [다시 시도] 비노출 (제안) — PM에게 별도 공유 |

**인수 조건**
- [ ] 메뉴 A: 프롬프트 입력 후 [생성하기] 클릭 시 배경 1장+캐릭터 다각도가 생성된다
- [ ] 메뉴 B: 자유 텍스트 또는 속성 선택 사이드바로 입력 후 [생성하기] 클릭 시 캐릭터 키이미지가 생성된다
- [ ] AI 프롬프트 변환 토글 On 상태에서 Enter/[생성하기] 클릭 시 영어 프롬프트 미리보기 없이 바로 생성이 진행된다
- [ ] 생성 중 로딩 상태가 표시되고 버튼은 비활성화된다
- [ ] 메뉴 A에서 선택한 비율이 생성 결과에 반영된다
- [ ] 빈 프롬프트로는 생성 요청이 발생하지 않는다
- [ ] 실패 시 에러 메시지와 [다시 시도] 버튼이 표시된다

## 5. 방향별 레퍼런스 (A001) — 메뉴 A/B로 흡수 (폐기)

> v0.3까지는 A001을 §4와 별도인 A등급 전용 기능으로 유지했으나, v0.4에서 **메뉴 A/B(§4)로 완전히 흡수 처리**한다. v0.3 원문은 비교용으로 `2026-07-10_기능명세서_v0.3.md`에 그대로 남아있음.

**흡수 판단 근거**

| A001이 원래 하려던 일 | 지금 메뉴 A/B가 이미 하는 일 |
|------|------|
| 캐릭터·배경을 여러 방향으로 확보해 영상 작업의 앵커 이미지로 활용 | 메뉴 A: 캐릭터 다각도(정면·측면·후면) + 배경 1장 |
| 방향별 세트를 완성도 있게 정리 | 메뉴 B: 턴어라운드+표정+소품+컬러 팔레트 캐릭터 키이미지 |

A001을 별도 기능으로 남겨두면 "이미 완료된 일을 07.23에 별도로 착수 판단"하게 되는 중복이 생기고, 마일스톤의 "A등급 = B등급 + A001" 조건도 사실상 이중 계산이 되어 등급 기준으로서 의미가 없어진다. 따라서 A001을 폐기하고 §4(메뉴 A/B)로 흡수한다.

**후속 조치 (팀 확인 필요)**

| 항목 | 조치 | 비고 |
|------|------|------|
| 마일스톤 v3.2 기능 ID 체계 | A001 표기 삭제 또는 "메뉴 A/B로 흡수됨" 각주 추가 | `framewave-docs` 통합 마일스톤 문서 반영 필요 |
| 등급 조건 (§3) | "A등급 = B등급 + A001" → "A등급 = B등급 + 메뉴 A/B(§4) 완성" 등으로 문구 조정 | 팀 확정 필요 |
| 07.23(목) A001 착수 판단 일정 | 별도 착수 판단 불필요 — 해당 슬롯은 §4 안정화 점검으로 대체 | 마일스톤 반영 필요 |

1. 클릭 시 이미지 생성 화면으로 이동, 프롬프트 입력창에 수정된 `prompt_en` 자동 입력
2. 이후 생성·저장은 M002~M004 플로우 재사용 — 저장 시 라이브러리 반영
3. 미저장 수정 내용이 있는 상태에서 화면 이탈 시 확인 없이 이동 (제안 — 프롬프트는 전달되므로 유실 없음)

> PRD v1(US-07)은 "동일 페이지 내 재생성" 기준이나, IA v0.2·기능명세서는 "이미지 생성 화면으로 이동" 기준. **v0.2는 화면 이동 방식으로 확정 명세** — PRD 업데이트 필요 (§11-5).

**인수 조건**
- [ ] [이미지 생성으로 이어서] 클릭 시 수정된 프롬프트가 이미지 생성 화면에 채워진다
- [ ] 업로드 → 추출 → 수정 → 재생성까지 5분 이내에 완료 가능하다 (서비스기획서 시나리오 1 성공 기준)

## 9. 내 프로필·설정 (AUTH-003~006)

| 항목 | 내용 |
|------|------|
| 화면 | 내 프로필·설정 화면 |
| API | BE 확정 (인증 방식 확정 후) |

### 9-1. 프로필 확인 (AUTH-003)

- 이메일, 가입일 표시 (읽기 전용)

### 9-2. 비밀번호 변경 (AUTH-004)

**입력 필드**

| 필드 | 필수 | 제약 |
|------|------|------|
| 현재 비밀번호 | ✅ | — |
| 새 비밀번호 | ✅ | 8자 이상 64자 이하 (제안), 현재 비밀번호와 달라야 함 (제안) |
| 새 비밀번호 확인 | ✅ | 새 비밀번호와 일치 |

**시스템 동작**

1. 현재 비밀번호 서버 검증 → 불일치 시 "현재 비밀번호가 일치하지 않습니다" 인라인
2. 변경 성공 → "비밀번호가 변경되었습니다" 토스트
3. 변경 후 재로그인 요구 여부 — BE 세션 정책에 따름 (BE 확정)

### 9-3. 로그아웃 (AUTH-005)

1. [로그아웃] 클릭 → 확인 모달 없이 즉시 세션 종료 (제안)
2. 로그인 화면으로 이동

### 9-4. 회원 탈퇴 (AUTH-006)

1. [회원 탈퇴] 클릭 → 확인 모달: 탈퇴 시 계정 복구 불가 안내 + **생성물(라이브러리) 처리 방침 문구 포함**
2. 생성물 처리 방침: 계정 삭제 시 라이브러리 생성물 일괄 삭제 (제안) — **정책 팀 확정 필요 (확인 필요)**
3. 탈퇴 완료 → 세션 종료 → 로그인 화면 이동 + "탈퇴가 완료되었습니다" 토스트
