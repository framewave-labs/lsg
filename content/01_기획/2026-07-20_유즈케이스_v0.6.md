## UC-IMAGE-01. 메뉴 A — 초기 레퍼런스 생성

| 항목 | 내용 |
|------|------|
| 유즈케이스명 | 메뉴 A — 초기 레퍼런스 생성 |
| 유즈케이스 ID | UC-IMAGE-01 |
| 주 액터 | 방송 PD/AD — 방송 프로그램 기획·제작 실무자. 포토샵 등 전문 디자인 툴이나 프롬프트 엔지니어링에 익숙하지 않음. 여러 프로그램을 동시에 맡는 경우가 많아 결과물을 하나하나 다듬기보다 촬영 전 짧은 시간 안에 캐릭터·배경의 대략적인 방향을 빠르게 잡아보고 싶어 함 |
| 목표 | 간단한 한 줄 프롬프트로 캐릭터·배경의 초기 형태를 빠르게 확인한다 |
| 사전조건 | 로그인 상태일 것 / 이미지 생성 화면(메뉴 A)에 진입해 있을 것 |
| 사후조건 | 배경 1장 또는 캐릭터 다각도 이미지가 결과 영역에 표시되고, 생성과 동시에 라이브러리에 자동 저장됨 |
| 우선순위 | P0 (B등급 필수 — M001~M004) |
| 관련 문서 | IA v0.6 화면구조(메뉴 A), 기능명세서 v0.8 S4-1~4-4 |

**기본 흐름**
1. 사용자가 메뉴 A 탭으로 진입한다
2. 프롬프트 입력창에 캐릭터 또는 배경에 대한 간단한 설명을 입력한다 (한국어/영어 자유)
3. 비율(1:1 기본)과 품질(medium 기본)을 선택한다
4. [생성하기] 클릭 또는 Enter
5. 시스템이 배경 이미지 1장 또는 캐릭터 다각도(정면·측면·후면) 이미지를 생성한다 — **예상 소요 시간 10~15초**, 이 구간은 스피너/스켈레톤 + "이미지를 생성하고 있습니다…" 문구로 대기 상태를 명확히 알려준다 (막연한 로딩만 보여주면 사용자가 멈춘 것으로 오인할 수 있음)
6. 결과가 결과 영역에 표시됨과 동시에 **별도 조작 없이 라이브러리에 자동 저장**된다
7. 사용자가 결과를 확인하고, 마음에 들면 [찜하기(하트)]로 즐겨찾기 표시하거나(UC-LIBRARY-03), 결과를 참고해 메뉴 B로 넘어간다

**대안/예외 흐름**
- 프롬프트가 빈값이면 [생성하기]가 비활성화 상태로 유지된다 (기본 흐름 4번 진입 불가)
- 생성 실패 시 에러 메시지와 [다시 시도]가 표시되고, 클릭 시 동일 입력값으로 재요청한다
- 생성 60초 초과 시 타임아웃 처리 후 [다시 시도] 노출 (예상 소요 시간의 4배 수준으로 여유를 둔 값)

## UC-IMAGE-02. 메뉴 B — 키이미지(캐릭터 시트) 생성

| 항목 | 내용 |
|------|------|
| 유즈케이스명 | 메뉴 B — 키이미지(캐릭터 시트) 생성 |
| 유즈케이스 ID | UC-IMAGE-02 |
| 주 액터 | 방송 PD/AD — 메뉴 A로 방향을 대략 잡은 뒤, 실제 촬영·제작 회의에 바로 쓸 수 있는 완성도 있는 캐릭터 자료가 필요한 실무자. 디자이너에게 의뢰할 시간 여유는 없지만, 표정·소품·컬러까지 구체적으로 정리된 레퍼런스가 있어야 다른 팀원(연출·미술)과 소통이 원활하다고 느낌 |
| 목표 | 메뉴 A에서 확인한 캐릭터 외형을 바탕으로, 실제 제작에 쓸 완성도 높은 캐릭터 시트(턴어라운드+표정+소품+컬러 팔레트)를 만든다 |
| 사전조건 | 로그인 상태일 것 / 이미지 생성 화면에서 메뉴 B 탭에 진입해 있을 것 (메뉴 A를 거치지 않고 바로 진입도 가능) |
| 사후조건 | 캐릭터 키이미지가 결과 영역에 표시되고, 생성과 동시에 라이브러리에 자동 저장됨 |
| 우선순위 | P0 (B등급 필수 — M001~M004) |
| 관련 문서 | IA v0.6 화면구조(메뉴 B), 기능명세서 v0.8 S4-1~4-4, 메뉴AB 프롬프트가이드 v1 S3 |

**기본 흐름**
1. 사용자가 메뉴 B 탭으로 진입한다
2. 입력 방식을 고른다 — **자유 텍스트**(PART1 서술+PART2 구조화 필드) 또는 **속성 선택 사이드바**(파트1 캐릭터 기본~파트5 의상·악세서리)
3-A. (자유 텍스트) PD 표준형 구조에 맞춰 프롬프트를 직접 작성한다
3-B. (속성 선택) 사이드바에서 캐릭터 속성·헤어·얼굴·의상 값을 순서대로 선택한다. 컬러 항목은 팔레트+헥스코드로 직접 지정
4. [생성하기] 클릭 또는 Enter
5. 시스템이 캐릭터 키이미지(턴어라운드+표정 배리에이션+소품 클로즈업+컬러 팔레트)를 생성한다 — 메뉴 A보다 담을 요소가 많아 **예상 소요 시간 15~20초** 수준으로 안내 (제안, 실측 후 조정)
6. 결과가 결과 영역에 표시됨과 동시에 **별도 조작 없이 라이브러리에 자동 저장**된다
7. 사용자가 결과를 확인한다 — 이후 영상 생성(UC-VIDEO-01)에서 이 캐릭터를 참고 레퍼런스로 활용, 마음에 들면 [찜하기(하트)]로 즐겨찾기 표시 가능(UC-LIBRARY-03)

**대안/예외 흐름**
- 자유 텍스트 경로에서 프롬프트가 빈값이면 [생성하기] 비활성화
- 속성 선택 사이드바 경로에서 필수 항목 미입력 시 [생성하기] 비활성화
- 생성 실패·타임아웃 시 UC-IMAGE-01과 동일한 에러 처리

## UC-PROMPT-01. AI 프롬프트 변환 토글 사용

| 항목 | 내용 |
|------|------|
| 유즈케이스명 | AI 프롬프트 변환 토글 사용 |
| 유즈케이스 ID | UC-PROMPT-01 |
| 주 액터 | 방송 PD/AD — 원하는 장면을 한국어로는 구체적으로 설명할 수 있지만, 영어 프롬프트 작성 경험이 거의 없어 이를 AI가 잘 이해하는 구조(구도·조명·스타일 키워드 등)로 바꾸는 데는 서투름. 별도로 프롬프트 작성법을 배울 여유는 없음 |
| 목표 | 한국어로 입력해도 AI가 이미지 생성에 최적화된 영어 프롬프트로 바꿔주도록 해, 프롬프트 엔지니어링 지식 없이도 생성 품질을 확보한다 |
| 사전조건 | 메뉴 A 또는 메뉴 B(자유 텍스트 입력 경로)에서 프롬프트 입력창에 텍스트가 입력되어 있을 것 |
| 사후조건 | 저장된 결과의 prompt 메타데이터가 변환 On 시 변환된 영문 프롬프트 기준으로 기록됨 |
| 우선순위 | P1 (필수 기능, P0 대비 우선순위만 낮음 — 디자이너·FE·BE 논의 완료, 구현 진행 중) |
| 관련 문서 | IA v0.6(전제조건 표), 기능명세서 v0.8 S4-1(공통 토글) |

**기본 흐름**
1. 사용자가 프롬프트 입력창에 한국어로 원하는 장면·캐릭터를 입력한다
2. "AI 프롬프트 변환" 토글을 On으로 켠다 (보조문구: "입력한 내용을 이미지 생성용 영어 프롬프트로 다듬어드려요")
3. [생성하기] 클릭 또는 Enter
4. 시스템이 백그라운드에서 Claude Sonnet으로 입력을 영어로 변환한다 — **변환된 영어 프롬프트는 화면에 노출하지 않는다**
5. 변환된 프롬프트로 바로 이미지 생성이 진행된다
6. 결과가 결과 영역에 표시된다

**대안/예외 흐름**
- 토글이 Off 상태면 입력한 텍스트를 그대로 생성 요청에 사용한다 (변환 단계 없음)
- 메뉴 B에서 속성 선택 사이드바 경로를 쓰는 경우 이 토글은 적용되지 않는다 (이미 구조화된 값이라 변환 대상 아님)
- 변환 자체가 실패해도 원본 입력 텍스트로 생성을 시도한다 (제안 — BE 협의 필요)

| 항목 | 내용 |
|------|------|
| 유즈케이스명 | 영상 생성 |
| 유즈케이스 ID | UC-VIDEO-01 |
| 주 액터 | 방송 PD — 프로그램 연출·기획을 총괄하며, 실제 촬영에 들어가기 전 장면의 톤·구도·움직임을 미리 확인하고 싶어 함. AI 영상 생성 툴 사용 경험은 많지 않고, 팀 예산을 직접 관리하는 경우가 많아 생성 비용에 민감함 |
| 목표 | 9컷 3x3 그리드 스토리보드 이미지와 영상 프롬프트로 짧은 영상 클립을 생성해 실제 촬영 전 장면을 미리 확인한다 |
| 사전조건 | 로그인 상태일 것 / 영상 생성 화면에 진입해 있을 것 / 9컷 3x3 그리드 이미지(예시 수령분 또는 자체 생성분, 번호 오버레이 없는 버전)를 보유하고 있을 것 |
| 사후조건 | 생성된 영상이 결과 영역에 표시되고, 완료와 동시에 라이브러리에 자동 저장됨. 오디오 포함 여부는 잠정 방침(두 모델 모두 ON 고정)이며 PD 공식 확인 대기 중 |
| 우선순위 | P0 (B등급 필수 — A002~A005) |
| 관련 문서 | 최종 IA v2.1 화면구조(영상 생성), 최종 기능명세서 v0.4 S4(영상 생성)·확인필요 #1·#4·#8, 최종 옵션값 확정 v0.10 S4(영상 생성 화면)·S4.3(모델별 전달 방식)·미해결 #6·#8, 최종 마일스톤 v3.3 |

> **(확인 필요 — BE 비교 테스트 진행 중)** 3x3 그리드 입력의 모델별 처리 방식은 2026-07-13 옵션값 확정 v0.10 S4.3에서 구체화됐으나 최종 확정은 아니다. **Seedance 2.0**은 그리드를 9장으로 슬라이스한 뒤 순서대로 레퍼런스 입력(`bytedance/seedance-2.0/reference-to-video`), 프롬프트는 `@Image1`~`@Image9` 참조 형식으로 변환하는 것이 확정 방향. **Kling 3.0**은 이미지 9장 동시 입력이 불가한 엔드포인트라 1번 컷=시작 프레임(`start_image_url`), 9번 컷=끝 프레임(`end_image_url`)으로 지정하고 그리드 전체를 레퍼런스로 추가 입력하는 현행 방식을 쓴다. 같은 스토리보드로 15초·720p 3건(① Seedance 그리드 1장 그대로 ② Seedance 9장 슬라이스 ③ Kling 현행 방식)을 비교 테스트해 순서 유지·컷 식별·전환 자연스러움을 판정 중 — 결과에 따라 아래 3번(모델 선택) 이후 흐름이 달라질 수 있다. 상세는 최종 옵션값 확정 v0.10 S4.3 참고.

**기본 흐름**
1. 사용자가 영상 생성 화면에 9컷 3x3 그리드 이미지 1장을 업로드한다 (B팀에서 실시간으로 전달받는 것이 아니라 예시로 수령한 이미지 또는 자체 생성분을 사용 — 자체 생성 시 컷 번호 오버레이가 영상에 노출될 수 있어 번호 없는 버전 사용)
2. 영상 프롬프트 1건을 입력한다 (좌상단=1번 → 우하단=9번 순서대로 scene/shot을 서술, 변환 없이 그대로 생성에 투입)
3. 모델을 카드 형태로 선택한다 — **Seedance 2.0 / Kling 3.0 둘 다 스토리보드 영상 생성 용도**이며 역할 분담은 없음(기존 "데모용/테스트용" 구분은 폐기, 2026-07-13 확정)
4. 영상 길이를 선택한다 — Kling 3.0은 3~15초, Seedance 2.0은 4~15초 범위(모델별 최소값 다름, 미지원 값은 UI 비활성화). 9컷을 다 담으려면 컷당 1.5~2초 확보가 전제라 실질적으로는 13~15초 구간을 주로 쓰게 됨
5. [영상 생성] 클릭 — 오디오 포함 여부는 잠정 방침(두 모델 모두 `generate_audio` ON 고정)이며, UI 토글은 미노출. PD 공식 확인 후 방침이 바뀌면 재조정될 수 있음
6. 시스템이 요청을 접수하고 상태를 `processing → completed`로 표시한다 (또는 `failed`) — 퍼센트 진행률은 사용하지 않고 **스피너 방식**으로 진행 표시(2026-07-13 PM·디자이너 협의 확정). **예상 소요 시간은 모델·길이에 따라 수십 초~수 분** 단위로 이미지보다 훨씬 길기 때문에, 스피너와 함께 경과 시간을 안내해 사용자가 멈춘 게 아니라는 걸 명확히 알려준다
7. 생성 완료 시 인라인 비디오 플레이어로 결과가 표시됨과 동시에 **별도 조작 없이 라이브러리에 자동 저장**된다
8. 마음에 들면 [찜하기(하트)]로 즐겨찾기 표시 가능(UC-LIBRARY-03)

**모델 선택 근거** (Kling 3.0 vs Seedance 2.0 — 2026-07-13 확정: 역할 분담 없음, 둘 다 스토리보드 생성 용도)

| 구분 | Kling 3.0 Standard | Seedance 2.0 |
|------|------|------|
| 해상도 | 1080p 고정 (v1은 Standard 티어만 노출, Pro·4K는 후속) | 사용자 직접 선택(720p·1080p, 4K는 미확인) |
| 단가 (오디오 OFF, 15초 기준) | $0.084/초 → $1.26 | 720p $0.3034/초 → $4.55 / 1080p $0.682/초 → $10.23 |
| 단가 (오디오 ON) | $0.126/초 (+50%) — 잠정 방침으로 두 모델 다 ON 고정 시 적용 | 요금 동일(오디오 On/Off 무관) |
| 3x3 그리드 입력 방식 | 1번=시작 프레임 + 9번=끝 프레임 + 그리드 전체 레퍼런스 (현행 방식) | 9장 슬라이스 후 `@Image1~9` 참조 (확정 방향, 전처리 개발 필요) |
| 비율 | 16:9, 9:16, 1:1 | auto, 21:9, 16:9, 4:3, 1:1, 3:4, 9:16 (전부 지원) |
| 일일 생성 한도 (BE 안전장치, UI 미노출) | 10개/일 | 720p 이하 5개/일, 1080p 2개/일 |

> 이전 버전(v0.5)의 "Kling=반복 테스트용 기본값 / Seedance=최종 데모용"이라는 역할 분담은 2026-07-13 통합본 확정으로 폐기됨. 두 모델 다 같은 입력(그리드+프롬프트)을 받아 스토리보드 영상을 생성하며, 차이는 처리 방식·해상도·단가일 뿐 용도 구분은 없다. 3x3 그리드 비교 테스트(옵션값 확정 v0.10 S4.3) 결과에 따라 슬라이스 전처리 개발 여부·Kling 방식 채택 여부가 결정되면 본 표를 갱신한다.

**대안/예외 흐름**
- 프롬프트가 빈값이면 [영상 생성] 비활성화
- 일일 생성 한도 도달 시 "오늘 생성 가능 횟수를 초과했습니다" 안내와 함께 생성 요청 차단
- 10분 초과 시 타임아웃 에러 + [다시 시도] (새 요청으로 처음부터 재시작)
- 상태가 `failed`면 에러 메시지 + [다시 시도], 이전 입력값은 유지
- 화면 이탈 시에도 서버 측 생성·자동 저장은 계속 진행됨 — 저장을 위해 화면에 머무를 필요 없음

- 찜은 저장 액션이 아니므로, 찜을 해제해도 라이브러리에서 항목이 사라지지 않는다 (삭제는 별도 액션, UC-LIBRARY-02)

## UC-REVERSE-01. 역프롬프트 — 업로드→분석→수정→재생성

| 항목 | 내용 |
|------|------|
| 유즈케이스명 | 역프롬프트 — 업로드→분석→수정→재생성 |
| 유즈케이스 ID | UC-REVERSE-01 |
| 주 액터 | 방송 PD — 레퍼런스가 될 만한 이미지(다른 프로그램 스틸컷, 무드보드 등)는 자주 접하지만, 그 스타일을 말로 옮겨 프롬프트를 처음부터 작성하는 건 부담스러워 함. "이거랑 비슷하게" 정도의 직관적인 재현을 원함 |
| 목표 | 마음에 드는 레퍼런스 이미지 하나로, 비슷한 스타일의 새 이미지를 빠르게 재현한다 |
| 사전조건 | 로그인 상태일 것 / 역프롬프트 화면에 진입해 있을 것 / 참고할 레퍼런스 이미지 파일을 보유하고 있을 것(jpg/png/webp, 10MB 이하) |
| 사후조건 | 레퍼런스 이미지 기반으로 재생성된 새 이미지가 라이브러리에 자동 저장된 상태 — 목표는 업로드부터 재생성까지 5분 이내 완료 |
| 우선순위 | P2 (S등급 도전 과제) |
| 관련 문서 | 기능명세서 v0.8 S8, 마일스톤 v3.2 S3(등급 조건) |

**기본 흐름**
1. 사용자가 레퍼런스 이미지를 업로드한다 (드래그앤드롭 또는 파일 선택)
2. [프롬프트 추출] 클릭
3. 시스템(Claude Sonnet 멀티모달)이 이미지를 분석해 스타일·조명·구도·렌즈·색감·분위기 6개 항목과 영문 프롬프트(`prompt_en`)를 추출한다
4. 추출 결과가 태그 칩 형태로, 프롬프트는 에디터에 표시된다
5. 사용자가 프롬프트를 직접 수정한다 (원본 보기 토글로 추출 원본과 비교 가능)
6. [이미지 생성으로 이어서] 클릭
7. 수정된 프롬프트가 이미지 생성 화면(메뉴 A/B)에 자동으로 채워지고, 이후 UC-IMAGE-01/UC-IMAGE-02 흐름을 그대로 따른다
8. 생성된 이미지는 별도 조작 없이 라이브러리에 자동 저장된다

**대안/예외 흐름**
- 미지원 형식·10MB 초과 업로드 시 각각 에러 메시지 표시, 업로드 진행 불가
- 분석 실패 시 에러 메시지 + [다시 시도] (업로드 파일은 유지)
- [원본으로 되돌리기]로 수정 내용을 추출 시점으로 되돌릴 수 있다
