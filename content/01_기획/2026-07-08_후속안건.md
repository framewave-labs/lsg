## 안건 2. 메뉴 A·B 결과물 라이브러리 저장 구조

**배경**
현재 라이브러리(A006)는 이미지·영상을 저장하는 구조로 설계되어 있다. 메뉴 B(키이미지)는 단순 이미지 외에 캐릭터 설정 데이터(컬러 코드, 의상 정보 등)를 함께 저장할 필요가 있을 수 있다.

**검토 필요 항목**
- 이미지 파일만 저장할지, 캐릭터 설정 데이터도 함께 저장할지
- 저장 시 BE DB 스키마 변경 범위
- 라이브러리 화면에서 메뉴 A 결과물(배경·캐릭터 다각도)과 메뉴 B 결과물(캐릭터 시트)을 구분해서 표시할지 여부
- 현재 라이브러리 방향(히스토리) vs 차후 방향(좋아요·스크랩) 전환 시 구조 영향

**담당**: BE 검토 후 PM·UX/UI 협의
**우선순위**: 중 (A006 구현 전 결정 필요)

## 안건 6. 메뉴 B — 레퍼런스 이미지 기반 캐릭터 유형 자동 판단

**배경**
메뉴 B(키이미지)에서 사용자가 레퍼런스 이미지를 업로드하면, 이미지를 Vision AI로 분석해 캐릭터 유형(인간형·동물형·로봇형·수인형)을 자동 판단하고 해당 유형에 맞는 속성값을 자동 세팅하는 방안.

**흐름**
```
메뉴 B 레퍼런스 이미지 업로드
  → Claude Sonnet Vision으로 자동 분석
  → 캐릭터 유형 판단 (인간형 / 동물형 / 로봇형 / 수인형)
  → 해당 유형에 맞는 속성값 자동 세팅
  → 사용자는 수정만 하면 됨
```

**가능 여부 확인**
- Claude Sonnet: 기존 API 키 그대로 Vision 사용 가능 (추가 계약 불필요)
- OpenAI GPT-4o Vision: GPT Image 2와 동일 키 사용 가능
- → **Claude Sonnet 단일화 권장** (프롬프트 변환 레이어와 동일 API, BE 구조 단순)
- 비용: 이미지 1장당 약 $0.01~0.02

**리소스 판단**
| 파트 | 수준 | 내용 |
|------|------|------|
| 디자인 | 낮음 | 업로드 후 분석 중 로딩 + 자동 세팅 결과 표시 |
| FE | 낮음~중간 | 업로드 이벤트에 분석 API 연결 + 조건부 렌더링 |
| BE | 중간 | Claude Sonnet Vision 호출 추가 + 유형 판단 프롬프트 설계 |

**담당**: BE 검토 후 PM·UX/UI 협의
**우선순위**: 중 (메뉴 B 기본 구현 후 착수 가능)

**업데이트 필요 항목**

**담당**: PM
