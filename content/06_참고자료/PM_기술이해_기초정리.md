### 1. API가 뭔지 (개념만)

```
API = 두 서비스가 대화하는 창구

우리 서비스 → (요청) → GPT Image 2 API → (응답: 이미지) → 우리 서비스
```

| 용어 | 뜻 | A2팀 예시 |
|------|-----|---------|
| **요청(Request)** | 우리가 보내는 것 | 프롬프트, 모델 선택, 비율 옵션 |
| **응답(Response)** | API가 돌려주는 것 | 생성된 이미지, 영상 URL, 추출된 프롬프트 |
| **API 키** | API를 쓰기 위한 비밀번호 | 절대 외부 공개 금지 |
| **호출(Call)** | API를 한 번 사용하는 것 | 비용 발생 단위 |

**왜 알아야 하나:**
비용 산정, 기능 범위 논의, 팀원에게 요구사항 전달할 때 기본 언어가 됨

---

### 2. 동기 vs 비동기

```
동기(Sync):   요청 → 기다림 → 응답  (이미지 생성: 수십 초 대기)
비동기(Async): 요청 → 번호표 받음 → 나중에 결과 확인 (영상 생성: 수 분 소요)
```

**A2팀 기능별 구분:**

| 기능 | 방식 | 이유 |
|------|------|------|
| F001 이미지 생성 | 동기 | 수십 초 내 응답 가능 |
| F002 스토리보드 그리드 | 동기 | 이미지 생성과 동일 |
| F003 영상 생성 | **비동기** | 수 분 소요 → 기다리는 동안 다른 작업 |
| F004 역프롬프트 | 동기 | 텍스트 분석이라 빠름 |

**비동기 처리 흐름 (영상 생성):**
```
1. 생성 요청 → request_id(번호표) 받음
2. 주기적으로 "완료됐나요?" 확인 (polling)
3. 완료 시 영상 URL 수신
```

**왜 알아야 하나:**
"영상은 왜 버튼 누르면 바로 안 나와요?" 질문에 답할 수 있어야 함.
로딩 UI 설계, 사용자 시나리오 작성 시 반드시 고려해야 하는 개념

---

### 3. 토큰(Token)이 뭔지

```
AI 모델의 비용 단위 = 토큰

"안녕하세요" ≈ 5토큰
영어 단어 1개 ≈ 1~2토큰
이미지 1장 ≈ 1,000~2,000토큰
```

**핵심 원리:**
- API 비용은 "호출 횟수"가 아니라 **"토큰 수"** 로 계산됨
- 프롬프트가 길수록 → 토큰 많아짐 → 비용 증가
- 이미지·영상 입력도 토큰으로 환산되어 과금

**A2팀 적용:**
- F001·F002: 이미지 출력 토큰이 비용의 대부분
- F004 역프롬프트: 이미지 입력 토큰 + 텍스트 출력 토큰
- 프롬프트 가이드 설계 시 "필요한 정보만 담아 간결하게" 원칙 중요

---

### 4. 환경변수 (.env)

```
.env 파일 = API 키 같은 비밀값을 코드 밖에 따로 저장하는 파일

OPENAI_API_KEY=sk-xxxxxxx
GEMINI_API_KEY=AIxxxxxxx
FAL_KEY=xxxxxxx
ANTHROPIC_API_KEY=sk-ant-xxxxxxx
```

**절대 하면 안 되는 것:**
- API 키를 코드에 직접 쓰는 것 → GitHub에 올라가면 외부에 노출
- .env 파일을 GitHub에 올리는 것 → 비용 폭탄, 보안 사고

**PM이 알아야 하는 이유:**
- 매니저에게 API 키 발급 요청할 때 이 개념으로 소통
- "env 세팅해야 해요"라는 말이 무슨 뜻인지 이해
- 팀원이 키를 공유할 때 안전한 방법(디스코드 DM 등)으로 전달하도록 안내

### 5. 프롬프트 엔지니어링 기초

PM이 직접 만들어야 하는 작업물입니다.

**좋은 이미지 프롬프트 구조:**
```
[스타일/화풍] + [주인공 묘사] + [장면 묘사] + [카메라/구도] + [분위기/조명]

예시:
"Cinematic drama style, Korean woman in her 30s with short black hair,
standing in a rainy alley at night, close-up shot showing tension,
dark and moody lighting, film noir atmosphere"
```

**9컷 그리드용 씬 분리 프롬프트 구조:**
```
씬 1: [배경 설명] + [앵글: wide shot] + [액션]
씬 2: [배경 설명] + [앵글: medium shot] + [액션]
씬 3: [배경 설명] + [앵글: close-up] + [감정]
...
```

**역프롬프트 출력 형식 (가안):**
```
Style: [화풍·그림체]
Lighting: [조명 방식]
Composition: [구도]
Lens: [렌즈·카메라]
Mood: [분위기·색감]
```

**A2팀 PM이 만들어야 하는 것:**
- 이미지 생성 프롬프트 가이드 (방송 제작자용 한국어 설명)
- 9컷 그리드용 씬 분리 프롬프트 템플릿
- 역프롬프트 출력 형식 정의

### 6. 프론트엔드 vs 백엔드

```
프론트엔드(FE): 사용자가 보는 화면
백엔드(BE):    눈에 안 보이는 서버 처리
```

**A2팀 기준:**

| 구분 | 담당 | 예시 |
|------|------|------|
| FE | 화면 | 프롬프트 입력창, 결과 이미지 그리드, 영상 플레이어, 로딩 표시 |
| BE | 서버 처리 | GPT Image 2 호출, Seedance 비동기 처리, Supabase 저장 |

> **B팀은 BE 0명** → 이미지 업로드·캔버스 조작이 프론트 중심이라 가능한 구성

---

### 7. Supabase 역할 구분

```
Supabase = F005 미디어 라이브러리의 저장소
```

| 구성 요소 | 역할 | 비유 |
|----------|------|------|
| **Storage** | 실제 이미지·영상 파일 저장 | 구글 드라이브 |
| **Database** | 파일 정보·메타데이터 저장 | 엑셀 시트 (파일명, 생성일, 프롬프트 등) |
| **Auth** | 로그인·인증 | MVP에서는 간단히 처리 |

---

### 8. 배포(Deploy)와 데모 URL

```
로컬(Local):  내 컴퓨터에서만 돌아가는 상태 → 나만 볼 수 있음
배포(Deploy): 인터넷에 올린 상태 → 누구나 URL로 접근 가능
```

**우리 팀 배포 방식:**
- Vercel 사용 → `https://kx-genstudio.vercel.app` 같은 URL 자동 생성
- GitHub에 코드 올리면 Vercel이 자동으로 빌드·배포

**PM이 알아야 하는 이유:**
중간점검·최종 발표에 **"배포 데모 URL"** 이 필수 제출 항목

---

### 9. GitHub 기초 (코드 말고 개념만)

```
GitHub = 코드 저장소 + 팀 협업 도구
```

| 용어 | 뜻 |
|------|-----|
| **Repository(레포)** | 프로젝트 전체 폴더 |
| **Commit** | 변경 사항을 저장하는 행위 |
| **Branch** | 독립적인 작업 공간 (실험해도 메인에 영향 없음) |
| **README** | 프로젝트 설명서 (PM이 작성) |

**PM이 알아야 하는 이유:**
- 최종 제출물에 **GitHub Public Repo** 필수
- README 작성은 PM 담당 → 서비스 설명, 실행 방법, 팀원 소개 등

### 10. 비용 모니터링 방법

각 API 대시보드에서 사용량 직접 확인 가능. **PM이 주 1회 이상 확인** 권장.

| API | 대시보드 접속 위치 |
|-----|----------------|
| OpenAI (GPT Image 2) | platform.openai.com → Usage |
| Google (Gemini) | aistudio.google.com → 사용량 |
| fal.ai (Seedance·Kling) | fal.ai → Dashboard |
| Anthropic (Claude) | console.anthropic.com → Usage |

**모니터링 체크 포인트:**
- 예산($150) 대비 현재 사용량
- 영상 API(fal.ai)는 특히 주의 — 비용이 가장 빠르게 쌓임
- 이상 급증 시 즉시 팀·매니저에게 공유

## PM이 몰라도 되는 것

- 코드 문법 (TypeScript, Python 등)
- API Route 구현 방법
- Supabase 쿼리 작성
- Docker, CI/CD, 서버 인프라 설정
