## 2. 핵심 결정사항 (요약)

1. **올인원 플랫폼은 이번 범위에서 제외(패스).** 각 팀이 A/B/C 기능을 **개별 프로젝트**로 개발한다. 통합은 향후 고도화 단계로 미룬다.
2. **단, UI/UX 디자인 시스템(폰트·컬러·디자인셋)은 전 팀 통일.** 나중에 합칠 때를 대비. 레퍼런스 제공과 최종 결정은 PD가 한다.
3. **A 라인이 사실상 메인 엔진**임을 재확인(이미지·영상 생성 + 역프롬프트). A↔B 연계가 많고 A가 과중하다는 점에 모두 공감.
4. **B 라인은 스토리보드 빌더**로 한정 — 키이미지+프롬프트 입력 → 9컷 아웃컷 이미지 + 샷별 프롬프트(3000자 이내) 생성, PDF/이미지 export, 캔버스(정리 공간).
5. **C 라인은 완전 후반작업** — 편집 완료 영상에 오디오(자막·효과음)를 덧입히는 단계.
6. **API 비용은 회사 통합 계정으로 일괄 결제.** 각 팀이 필요 API/예상 금액을 정리해 매니저에게 전달.
7. **근태·근로계약서 운영 시작** — 내일(06-24)부터 09:30 출근 / 12:30 퇴근 체크 필수, 근로계약서는 오늘 중 제출.

## 3. 프로젝트 범위·역할 정의 (회의 확정 버전)

> 회의 내내 A/B/C 경계에 혼선이 있었고, 아래로 정리·합의됨. 명세서의 라인 구분과 일치.

| 라인 | 역할 | 입력 → 출력 | 비고 |
| --- | --- | --- | --- |
| **A 생성 엔진 (메인)** | 모든 **이미지 생성 tool** + **영상 생성 tool** + **역프롬프트** | 프롬프트 → 이미지/영상, 이미지 → 프롬프트(역추출) | 캐릭터 시트·360도·키이미지 생성 포함. 영상 API 사용 시 음성도 1차로 함께 생성됨. 작업량 과중 |
| **B 스토리보드** | 스토리보드 빌더 + 캔버스 | A의 키이미지 + 사용자 프롬프트(시나리오) → **9컷 아웃컷 이미지 + 샷별 프롬프트** | PDF=팀 공유용 / 이미지=Seedance 투입용. 프롬프트 합계 3000자 이내(Seedance 제약). 캔버스는 생성보다 **정리(배치) 공간**. 백엔드 비중 낮음 |
| **C 오디오 툴킷** | 텍스트→SFX + 오디오→자막 | 텍스트 → 효과음 / 오디오 → SRT·ASS 자막 | 완전 후반. 효과음 라이브러리는 사내 보유분 활용 가능(카테고리 분류는 PD가 1차 정리) |

**워크플로우 전체 흐름(PD 설명):** 기획(텍스트) → 스토리보드 9컷 → 샷별 프롬프트 → 첫 프레임(키이미지) 정밀 생성 → 영상화 → 합치기 → 음악·자막. 첫 프레임이 망가지면 뒤 영상도 망가지므로 키이미지를 디테일하게 뽑는 것이 중요.

## 4. 주요 논의 사항

### 4.1 올인원 vs 개별 프로젝트 (이번 회의 최대 쟁점)

- **PD 원안:** 이미지·영상·캔버스·오디오·텍스트 해설을 한 페이지에서 호환·연동하는 **올인원 플랫폼**. 궁극 목표는 Higgsfield/Magnific 같은 올인원 + 수익화.
- **현실 제약:** 기간 1개월, 가용 시간 주 15시간(전체 ~60시간), 인턴 수준. 처음부터 통합으로 시작하면 코드가 방대해져 합치기 어렵고, 포트폴리오용으로도 무리.
- **합의:** **올인원 패스.** 기능별 개별 프로젝트로 진행 → 각자 완주 → 향후(2~3개월 추가 시) 별도 인력이 통합 가능. 개별 진행이 인턴들 포트폴리오에도 유리.
- **단서 조항:** **디자인 시스템만 처음부터 통일**(동일 UI/UX) → 나중에 합칠 여지를 남김.

### 4.2 디자인 / UI·UX

- PD가 **레퍼런스를 먼저 공유**, 의견 수렴 후 **최종 결정은 PD**가 함. PD 본인은 "디자인은 잘 모른다"며 형광 톤·튀는 색 선호.
- CI 폰트 제공 가능하나 **CI로 강제하지 않기로** 함(다양한 레퍼런스에서 선택).
- 각 팀 UX/UI 담당자와 PD가 폰트·글꼴·컬러·디자인셋 수준에서 조율. (세세한 크기 등은 담당자와 PD가 직접 결정)

### 4.3 팀 구성 / 인원 배분

- A가 무거워 인력 재배치 고민(A로 더 몰까 / A 통합 시 10명 초과 우려). → 일단 **현 구성대로 진행 후 조정**.
- **B는 백엔드 0명** 구성 — 이미지 변환 등 가벼운 작업이라 Claude로 구현 가능하나, "두 팀당 1명 정도 있으면 좋겠다"는 의견. (확답 보류)
- A1/A2, B1/B2는 **각 팀이 같은 프로젝트를 별도로** 개발 → 일종의 경쟁/서바이벌, 좋은 기능은 추후 취합. 구현 방식이 달라 다양한 결과를 볼 수 있다는 장점.

### 4.4 기술·모델 관련 요청

- **이미지 메인 모델:** 현재 명세서는 Gemini. PD는 **GPT 이미지(gpt-image)를 메인으로** 갈 수 있는지 검토 요청(현재 이미지 품질 원탑으로 인식). 매니저: 곧 확인.
- **프롬프트 생성:** Claude가 영문 프롬프트를 가장 잘 뽑음 → B 스토리보드는 **Claude API** 사용. 프롬프트는 순서(그림체→톤→…)가 있고 텍스트 형식이면 충분.
- **캐릭터/배경 일관성 문제:** 캐릭터 시트·배경 시트를 다각도로 뽑을 때 일관성 유지가 안 되는 게 가장 불편. 단 이는 생성모델(Gemini/GPT) 자체 한계 → 프론트에서 충돌을 줄이는 방향으로 보완 모색.
- **역프롬프트:** 원하는 톤의 이미지를 못 뽑을 때, 참조 이미지를 넣으면 그 이미지를 재현하는 프롬프트를 역으로 추출하는 기능(A 라인, 독립 기능).
- **로컬(ComfyUI) 방식은 제외** — 고사양 PC·상시 서버 부담으로 API 끌어오는 방식으로 결정.

### 4.5 API 비용 / 결제

- **회사 통합 계정으로 일괄 결제** 후 공유. 각 팀이 "우리는 무엇이 필요하다"를 정리해 전달 → 매니저가 취합 타이밍에 일괄 결제(경영지원팀 품의 절차).
- 1차 대상: **OpenAI, Claude, Seedance, Kling** 등.
- **영상 비용이 최대** — 토큰 예측이 어려움. 각 팀이 **대략 예상 금액**도 함께 정리해 전달.

### 4.6 운영 / 소통

- **디스코드:** 직무별·소규모 채널 + 음성 채널 운영. 매니저가 방 생성, 전원 초대.
- **카톡:** PM 소통 채널 별도 운영. PD는 전화 수신이 어려워 **톡으로 남기면 확인**.
- 분위기: "싸우지 말고 즐겁게, 안 되는 건 빨리 공유" — 막히면 PD/매니저가 정리·포기 판단.

### 4.7 바이브 코딩 (참고)

- 매니저는 **Claude Code(CLI, Opus) 메인 사용**, 손코딩 거의 없음. 백엔드 없는 프론트 서비스는 AI 프롬프팅만으로 구현 가능, API 연동부터는 학습 필요.
