## 1. 한국어 프롬프트 처리 방식

### 1.1 배경

GPT Image 2는 영어 프롬프트에 최적화되어 있다. 그러나 A2팀 서비스 사용자는 방송 제작자로, 한국어로 기획 의도를 입력하는 것이 훨씬 자연스럽다. 이 간극을 메우기 위해 **Claude Sonnet을 프롬프트 변환 레이어로 활용**한다.

### 1.2 처리 흐름

```
사용자 (한국어 입력)
    ↓
Claude Sonnet
  · 한국어 → 영어 변환
  · 구도·조명·분위기·스타일 등 시각 요소 구체화
  · GPT Image 2 최적화 포맷 적용 (80단어 이내, photorealistic 끝맺음)
    ↓
GPT Image 2 (최적화된 영어 프롬프트로 생성)
    ↓
이미지 결과
```

**단순 번역이 아니라 "최적화 변환"이 핵심이다.**

| 사용자 입력 (한국어) | Claude 변환 결과 (영어) |
|-------------------|----------------------|
| 빗속에서 우산 들고 걷는 주인공 | `the protagonist walking through a narrow alley in heavy rain, holding an umbrella, medium shot, dramatic side lighting reflecting off wet pavement, cold blue tones, tense mood, Korean drama style, cinematic, photorealistic, high detail` |
| 황혼 무렵 한강변을 걷는 노부부 | `elderly couple walking along the Han River at dusk, holding hands, wide shot, warm golden hour lighting, silhouette against orange sky, peaceful and nostalgic mood, cinematic, photorealistic, high detail` |

### 1.3 UX 방식 (반자동 — 권장)

사용자가 한국어로 입력하면 Claude가 변환한 **영어 프롬프트를 FE에 노출**하고, 사용자가 수정 후 생성하는 방식을 권장한다.

```
[한국어 입력창]
  "빗속에서 우산 들고 걷는 주인공"

  ↓ 변환됨 (Claude Sonnet)

[영어 프롬프트 미리보기 · 수정 가능]
  "the protagonist walking through a narrow alley..."

  → [이미지 생성] 버튼
```

**이 방식을 권장하는 이유**
- 사용자가 변환 결과를 확인하고 수정할 수 있어 생성 품질 통제 가능
- 역프롬프트(S등급)의 `prompt_en` 노출 방식과 UX 일관성 유지
- 사용자가 자연스럽게 영어 프롬프트 패턴을 학습하게 됨

### 1.4 Claude Sonnet 시스템 프롬프트 초안 (이미지)

```
You are a professional prompt engineer specializing in AI image generation.

When given a Korean description, convert it into an optimized English prompt for GPT Image 2.

Rules:
- Keep the original intent and subject intact
- Add specific visual elements: shot type, camera angle, lighting, color tone, mood, style
- End every prompt with: photorealistic, high detail
- Keep under 80 words
- Output only the English prompt — no explanation, no Korean

Example:
Input: 빗속에서 우산 들고 걷는 주인공
Output: the protagonist walking through a narrow alley in heavy rain, holding an umbrella, medium shot, dramatic side lighting reflecting off wet pavement, cold blue tones, tense mood, Korean drama style, cinematic, photorealistic, high detail
```

> 이 시스템 프롬프트는 BE 구현 시 조정 가능. 현재는 초안. 영상용 시스템 프롬프트는 4.5 참고.

## 2. 이미지 프롬프트 작성 원칙

### 2.1 기본 구조 (순서 중요)

```
[피사체 + 동작]  →  [구도/샷 타입]  →  [조명]  →  [분위기/색감]  →  [스타일]  →  [품질 지시어]
```

**예시:**
```
A woman in a hanbok standing in a palace courtyard,    ← 피사체
wide shot, slightly high angle,                        ← 구도
golden hour sunlight, soft shadows,                    ← 조명
warm amber tones, nostalgic mood,                      ← 분위기/색감
cinematic, Korean period drama style,                  ← 스타일
photorealistic, high detail, 4K                        ← 품질
```

### 2.2 필수 끝맺음 (모든 프롬프트에 포함)

```
photorealistic, high detail
```
→ GPT Image 2는 이 두 단어가 있을 때 결과물 품질이 현저히 올라갑니다.

### 2.3 피해야 할 것

| 하지 말 것 | 이유 |
|-----------|------|
| 한국어로 직접 GPT Image 2에 입력 | Claude Sonnet 변환 레이어를 반드시 거칠 것 |
| 너무 긴 프롬프트 (80단어 초과) | 후반부 키워드 반영률 하락 |
| 추상적 감정만 적기 ("sad" 만) | 구체적 시각 요소로 변환 필요 ("tears on cheek, drooping shoulders") |
| 유명인 이름 사용 | API 정책 위반 가능 |
| 텍스트/글씨 요청 | 이미지 내 텍스트 품질이 불안정함 |

## 4. 영상 프롬프트 작성 원칙

### 4.1 기본 구조

v2에서는 엔진이 정해지지 않은 상태로 범용 구조만 다뤘으나, 영상 엔진이 **Kling 3.0 / Seedance 2.0**으로 확정되면서 두 엔진이 공유하는 공통 뼈대에 엔진별 최적화를 얹는 구조로 확장한다.

```
[SCENE 선언 (샷 수 / 길이 / 화면비)]
    ↓
[피사체 + 동작]  →  [카메라 무브먼트]  →  [배경/환경]  →  [조명/시간대]  →  [분위기]  →  [스타일]
```

**예시:**
```
A woman slowly walks along a riverside path,          ← 피사체 + 동작
camera gently tracking her from behind,               ← 카메라 무브먼트
autumn leaves falling, golden foliage on both sides, ← 배경
warm golden hour sunlight,                            ← 조명
melancholic and peaceful mood,                        ← 분위기
cinematic, Korean drama style                         ← 스타일
```

### 4.2 영상 프롬프트 특이사항 (공통)

- **동작을 구체적으로 묘사할수록 결과가 좋습니다**
  - 나쁜 예: `a person walking`
  - 좋은 예: `a woman slowly walks, slightly hunched shoulders, looking down at her phone`
- **카메라 무브먼트를 반드시 포함하세요** — 없으면 대부분 정적 샷으로 나옵니다
- **카메라 무브먼트는 2~3개 이내로 조합**, "+"나 "while"로 연결 (예: `crane up + slow pan right`) — 3개 초과 시 두 엔진 모두 결과가 불안정해짐
- **손·소품 등 신체 접촉이 있는 동작은 반드시 대상 사물을 명시** — "she moves her hands" (X) → "her fingers firmly grip the edge of the ceramic cup" (O)
- **5초 테스트 클립은 단순한 동작 1가지만** — 복잡한 장면은 10초+로
- **한국어 입력 시** 이미지 생성과 동일하게 Claude Sonnet 변환 레이어를 거칩니다 (엔진 선택에 따라 4.5의 분기 규칙 적용)

### 4.3 Kling 3.0 최적화

**구조 원칙**
Kling 3.0은 "감독의 지시문"처럼 읽히는 프롬프트에 가장 잘 반응한다. 권장 순서:
```
Scene → Characters → Action → Camera → Audio & Style
```

**멀티샷 스토리보드**
- 최대 **6샷**을 하나의 출력에 담을 수 있음(A004 등 시퀀스형 기능에 활용 가능)
- 여러 샷을 쓸 경우 각 샷을 명확히 라벨링하고, 샷별로 프레이밍·피사체·동작을 분리 기술
```
SHOT 1: [프레이밍] — [피사체+동작] — [카메라]
SHOT 2: [프레이밍] — [피사체+동작] — [카메라]
...
```

**캐릭터 일관성**
- 동일 캐릭터를 여러 샷/여러 영상에서 쓸 경우 **레퍼런스 이미지를 고정**하고 캐릭터 묘사 문구를 **토씨 하나 안 바꾸고 반복** — 단어 순서나 표현이 조금만 달라져도 다른 캐릭터로 해석될 수 있음
- 메뉴 B 캐릭터 시트(예: "미라 한" — [[2026-07-08_메뉴AB_프롬프트가이드_v1]] 3.3)의 SUBJECT/CLOTHING/ACCESSORIES 필드 문구를 그대로 영상 프롬프트에 복붙하는 방식을 권장

**네이티브 오디오·대사**
- 대사·환경음·음성 톤 지정을 프롬프트 안에서 지원
- 오디오를 쓸 경우 **누가, 언제 말하는지** 명시 (특히 멀티 캐릭터 씬에서 필수)
```
Audio: [캐릭터명]이 "[대사]"라고 말함, [톤/감정] / 배경음: [환경음 종류]
```

**자주 하는 실수**

| 하지 말 것 | 이유 |
|-----------|------|
| 손동작을 허공에서 처리 ("moves her hands") | 어색한 움직임으로 렌더링됨 — 반드시 사물에 앵커링 |
| 여러 샷을 한 문단에 압축 | 요소가 뒤섞여 의도와 다른 결과 |
| 캐릭터 묘사 문구를 매번 다르게 표현 | 동일 캐릭터인데 외형이 미묘하게 달라짐 |
| 카메라 무브먼트 생략 | 대부분 정적 샷으로 출력 |

### 4.4 Seedance 2.0 최적화

**구조 원칙**
프롬프트 **최상단에 메타 정보(샷 수 / 총 길이 / 화면비)를 먼저 선언**하고, 이후 카메라·캐릭터(정체성)·동작을 **완전히 분리된 블록**으로 작성해야 한다. 세 요소를 한 문장에 섞으면 프레이밍 불안정·얼굴 변형·동작 붕괴가 발생하기 쉽다.

```
META: [샷 수] shots, [총 길이]초, [화면비]

IDENTITY: [캐릭터 정체성 — 외형·의상 고정 묘사]
CAMERA: [카메라 무브먼트만]
ACTION: [피사체 동작만]
STYLE: [조명/분위기/스타일]
```

**카메라 무브먼트 표현**
- **정밀한 용어 사용** — "move the camera" (X) → "smooth dolly forward", "slow pan left" (O)
- **리듬감 있는 언어**로 서술 — 카메라 스펙(24fps, f/2.8 등) 대신 "slow, smooth, stable, gradual, gentle" 같은 편집자에게 말하듯 한 표현 사용
- **안정화 키워드** 추가 — `stabilized`, `gimbal shot`, `steadicam` 등으로 흔들림 없는 결과 유도
- **타이밍 명시** — "3-second slow zoom in on the subject's eyes", "rapid 1-second whip pan to the left"처럼 구간·속도를 숫자로 지정

**자주 하는 실수**

| 하지 말 것 | 이유 |
|-----------|------|
| 샷 수/길이/화면비 선언 생략 | 의도한 길이·비율로 나오지 않음 |
| 카메라·캐릭터·동작을 한 문장에 혼합 | 프레이밍 불안정, 얼굴 변형 |
| 카메라 스펙(mm, fps, ISO) 나열 | Seedance는 기술 스펙보다 리듬 언어에 더 잘 반응 |
| 무브먼트 4개 이상 조합 | 모델이 전부 실행하지 못해 결과가 어색해짐 |

### 4.5 엔진 선택 및 한국어 입력 처리

- 한국어 입력 → 영어 변환은 1.2의 **Claude Sonnet 변환 레이어**를 그대로 사용하되, 시스템 프롬프트에 **엔진 분기**를 추가해야 함 (사용자가 Kling 3.0 / Seedance 2.0 중 선택 → 변환 규칙 분기)
- FE 노출 방식도 1.3(반자동 — 변환 결과 미리보기 후 수정 가능)과 동일하게 유지 권장

**엔진별 변환 시스템 프롬프트 분기 예시**
```
[공통] 사용자 의도·피사체를 유지하며 시각 요소(구도/조명/분위기)를 구체화

[Kling 3.0 선택 시]
- Scene → Characters → Action → Camera → Audio & Style 순서로 출력
- 멀티샷일 경우 SHOT 1/2/3... 라벨링
- 손동작은 반드시 사물에 앵커링

[Seedance 2.0 선택 시]
- 최상단에 META(샷 수/길이/화면비) 블록 먼저 출력
- IDENTITY/CAMERA/ACTION/STYLE 블록 분리
- 카메라 무브먼트는 리듬감 언어 + 안정화 키워드 포함
```

> 이 분기 로직은 BE 구현 시 조정 가능. 현재는 초안.

## 8. 품질 판단 기준

2주차 이미지 생성 완성 시 아래 기준으로 판단합니다. 영상 품질 기준은 시연 시나리오 확정 후 별도 추가 예정.

### Pass 기준 (최소)

| 항목 | 기준 |
|------|------|
| 프롬프트 반영도 | 핵심 피사체·분위기가 이미지에 반영됨 |
| 해상도 | 흐릿하지 않고 1024px 이상 |
| 구도 | 주요 피사체가 잘린 경우 없음 |
| 생성 성공률 | 5회 시도 중 3회 이상 Pass 기준 충족 |

### Good 기준 (목표)

| 항목 | 기준 |
|------|------|
| 조명·색감 정확도 | 프롬프트에 명시한 조명 방향·색감이 표현됨 |
| 분위기 일치 | 방송 제작 맥락에서 바로 쓸 수 있는 퀄리티 |
| 프롬프트 없이도 인식 | 이미지만 봐도 장르·상황이 명확히 전달됨 |
| Claude 변환 품질 | 한국어 입력 의도가 영어 프롬프트에 정확히 반영됨 |

### 평가 방법

- PM 이슬기 or 손찬용이 테스트 프롬프트 세트 5~10개로 생성 시도
- **한국어 입력 → Claude 변환 → GPT Image 2 생성** 전체 흐름으로 테스트
- GPT Image 2 결과물 품질 기록 (Pass/Good 기준 체크)
- 품질 좋은 결과는 라이브러리에 저장해 데모용으로 보관
